# /blog-review

블로그 글을 메타데이터 검증, 논리성, 흐름, 근거, 가독성 관점에서 개선합니다.

**사용법**: `/blog-review {slug}`

예시: `/blog-review about-neural-network`

---

## 역할

당신은 시니어 개발자이자 테크 블로그 에디터입니다.
주어진 글을 "메타데이터 정확성, 논리성, 흐름, 근거, 가독성" 관점에서 개선합니다.

## 실행 순서

### 0단계: 브랜치 생성 및 파일 로드

**사용자 확인 없이 자동으로 실행합니다.**

```bash
git checkout -b review/$ARGUMENTS
```

슬러그: `$ARGUMENTS`
대상 파일: `content/blog/$ARGUMENTS/index.mdx`

`content/blog/$ARGUMENTS/index.mdx`를 읽고 전체 내용을 파악합니다.
`content/blog/$ARGUMENTS/` 디렉토리의 파일 목록도 확인합니다.

---

### 1단계: 메타데이터 & 자산 검증

다음 항목을 **자동으로 검사**하고 결과 리포트를 출력합니다.

#### 1-1. Tags 네이밍 검증

tags 필드의 각 태그를 아래 규칙으로 검사합니다:

**변환 규칙 (순서대로 적용):**

| 처리 | 규칙 | 예시 |
|------|------|------|
| 소문자화 | 모든 대문자 → 소문자 | `Redis` → `redis` |
| 공백 → 하이픈 | 공백 한 칸 이상 → `-` | `Distributed File System` → `distributed-file-system` |
| 언어 특수문자 | `C++` → `cpp`, `C#` → `csharp`, `.NET` → `dotnet` | 의미 보존 우선 |
| 점(.) 제거 | `.` 제거 후 앞뒤 합침 | `Node.js` → `nodejs` |
| 나머지 특수문자 | 알파벳·숫자·하이픈 외 제거 | `OAuth2.0` → `oauth20` |
| 연속 하이픈 정리 | `--` 이상 → `-` | `rate--limit` → `rate-limit` |
| 앞뒤 하이픈 제거 | 선행/후행 `-` 제거 | `-tag-` → `tag` |

각 태그에 대해 현재값과 변환 후 값을 비교하여 수정이 필요한 태그 목록을 표시합니다.

#### 1-2. 이미지 파일 존재 확인

index.mdx 본문에서 이미지 참조를 모두 추출합니다:
- Markdown 형식: `![alt](./image.png)`
- JSX 형식: `src="./image.png"`, `image: "./image.png"`
- 프론트매터의 `image:` 필드

각 참조된 이미지 파일이 `content/blog/$ARGUMENTS/` 디렉토리에 실제로 존재하는지 확인합니다.

**리포트 형식:**
```
[이미지 검사]
✓ ./nn.png - 존재
✓ ./neuron.png - 존재
✗ ./missing-image.png - 파일 없음 (참조 위치: 3번째 이미지)
```

#### 1-3. Title & Description 검토

현재 title과 description을 분석하여 개선안을 제안합니다:

**Title 검토 기준:**
- 제목은 원문의 제목을 그대로 번역한다. 원문에 충실하게 번역한다. 장황하더라도 요약하지 않고, 의역보다 직역을 우선하되 자연스러운 한국어로 표현한다.
- 글의 핵심 주제를 명확히 표현하는가?
- 독자의 관심을 끌 수 있는가?
- 너무 길거나 짧지 않은가? (권장: 30~60자)
- 시리즈 글인 경우 순서 표기가 명확한가?
- **콜론 `:` 포함 여부 확인** — 포함 시 반드시 제거 (Gatsby 빌드 오류 원인)

**Description 검토 기준:**
- 글 전체 내용을 1~2문장으로 압축하는가?
- SEO에 효과적인 키워드가 포함되어 있는가?
- 독자가 읽기 전에 무엇을 얻을지 알 수 있는가?
- 너무 길거나 짧지 않은가? (권장: 80~160자)

**검토 결과 형식:**
```
[Title 검토]
현재: "TensorFlow를 통해 딥러닝의 기반 기술인 NN(Neural Network) 이해(1)"
평가: 길이 적절, 키워드 명확
개선안: 없음 (또는 구체적인 대안 제시)

[Description 검토]
현재: "딥러닝의 기반 기술인 뉴럴 네트워크에 대해..."
평가: 160자 초과, 내용 나열식
개선안: "TensorFlow 2.0과 Keras를 이용해 뉴럴 네트워크의 핵심 개념(데이터 준비, 문제 유형, 전처리, 모델 정의)을 코드와 함께 단계별로 설명합니다."
```

**1단계 검증 결과 전체를 사용자에게 보여주고 계속 진행할지 확인합니다.**

수정이 필요한 항목이 없으면 사용자에게 알리고 2단계로 넘어갑니다.

---

### 2단계: 콘텐츠 문제 진단

아래 기준으로 본문 내용을 분석합니다:

**분석 항목:**
- 논리 구조: 흐름이 자연스러운가? 논리적 점프가 있는가?
- 맥락 부족: 독자가 이해하기 위한 배경 설명이 충분한가?
- 근거 부족: 주장과 증거(데이터, 예시, 인용)의 균형
- 기술 설명 부족: 기술 개념이 충분히 설명되었는가?
- 가독성: 문장 길이, 단락 구성, 제목 체계

각 항목별로 다음을 명시합니다:
- 문제 위치 (섹션명 또는 단락)
- 왜 문제인지
- 어떻게 개선해야 하는지

**분석 결과를 사용자에게 보여주고 다음 단계 진행 여부를 확인합니다.**

---

### 3단계: 개선 계획

글을 더 좋게 만들기 위한 계획을 수립합니다:

- 전체 구조 재설계 (섹션 순서, 추가/삭제할 섹션)
- 추가해야 할 내용 (구체적으로 어떤 내용을)
- 제거/축소할 내용 (왜 제거하는지 이유 포함)
- 강조할 핵심 메시지

**계획을 사용자에게 보여주고 다음 단계 진행 여부를 확인합니다.**

---

### 4단계: 재작성

위 모든 단계의 검토 결과를 반영하여 `content/blog/$ARGUMENTS/index.mdx`를 수정합니다.

**수정 항목:**
1. Tags: 네이밍 규칙 위반 태그 자동 수정
2. Title: 1단계 개선안이 있으면 반영 (사용자가 승인한 경우)
3. Description: 1단계 개선안이 있으면 반영 (사용자가 승인한 경우)
4. 존재하지 않는 이미지 참조: 주석 처리하거나 사용자와 협의
5. 본문: 2~3단계 개선 계획 반영

**재작성 원칙:**
- 개발 블로그 스타일 유지 (기술적이고 명확하게)
- 불필요한 장황함 제거, 핵심 위주로 명확하게
- 기존 이미지 참조 경로 유지 (`./image.png`)
- 코드 블록은 언어 명시
- 한국어 유지
- `date` 필드는 수정하지 않음, `lastModificationTime`은 오늘 날짜로 업데이트

수정 완료 후 변경 내용 요약을 사용자에게 보여줍니다.

---

### 5단계: 커밋 및 PR 생성

**사용자 확인 없이 자동으로 실행합니다.**

```bash
git add content/blog/$ARGUMENTS/
git commit -m "docs: $ARGUMENTS 블로그 글 품질 개선"
git push -u origin review/$ARGUMENTS
```

PR 생성:
```bash
gh pr create \
  --title "[$ARGUMENTS] 블로그 글 품질 개선" \
  --body "..."
```

PR 본문에 포함할 내용:
- 메타데이터 수정 사항 (tags 정규화, title/description 개선)
- 이미지 검사 결과
- 콘텐츠 개선 사항 요약
- 개선 항목 체크리스트
