코딩 스타일 가이드는 보통 언어별로 존재한다. Python에는 PEP 8이 있고, Java에는 [Google Java Style Guide](https://google.github.io/styleguide/javaguide.html)가 있으며, Go에는 Effective Go가 있다. Rust 프로젝트를 시작하면 `rustfmt`와 Clippy가 스타일을 잡아준다. 이렇게 언어별 컨벤션은 잘 정비되어 있지만, 막상 여러 언어를 오가며 작업할 때 일관된 설계 철학이 없으면 코드는 언어만 다를 뿐 같은 문제를 반복한다.

LLM을 활용해 Java, Rust, TypeScript, Python, Go 등 여러 언어로 동시에 작업하면서 이 문제가 더 선명해졌다. AI가 코드를 생성할 때 적용할 수 있는 **언어 독립적인 설계 원칙**이 필요했다. 단순히 "중첩을 줄여라", "이름을 명확히 해라" 수준이 아니라, 왜 그렇게 해야 하는지 근거가 있는 원칙을 기술한다. 위임을 잘하려면 기준이 명확해야 한다.

## 철학적 뿌리

이 코딩 스타일은 세 사람의 사상을 중심으로 구성된다.

**Kent Beck**은 XP(Extreme Programming)와 TDD를 통해 "동작하는 코드를 빠르게, 그리고 지속적으로 개선하라"는 관점을 제시했다. 그의 핵심 통찰은 소프트웨어 설계의 목표가 완벽한 구조를 한 번에 만드는 것이 아니라, **변경하기 쉬운 상태를 항상 유지하는 것**이라는 점이다.

**Martin Fowler**는 리팩토링을 체계화했다. 그에게 리팩토링은 기능을 추가하기 전에 코드를 정돈하는 별도의 단계가 아니다. 코드를 읽고 이해하고 수정하는 모든 과정에 자연스럽게 녹아 있어야 하는 **개발 리듬**이다.

**Eric Evans**는 DDD(Domain-Driven Design)를 통해 코드가 비즈니스 도메인의 언어를 반영해야 한다고 주장했다. 코드베이스가 도메인 전문가와 개발자가 함께 쓰는 언어로 작성될 때, 요구사항의 변화를 코드에 정확하게 반영할 수 있다.

이 세 관점은 겉으로는 다르지만 하나의 방향을 가리킨다. **코드는 읽히고, 이해되고, 변경될 것이다. 그 변경을 쉽게 만드는 것이 설계의 본질이다.**

## 개발 원칙 (작업 방식과 의사결정 기준)

다음 다섯 가지는 코드 구조나 언어보다 앞서는, 작업 방식과 의사결정 기준에 관한 원칙이다. 나는 코드를 작성하기 전에 항상 이 원칙들을 먼저 떠올린다.

### 1. 코드는 의도를 명확하게 말해야 한다

- **의미 있는 이름으로 의도를 표현한다**: 변수, 함수, 클래스, 모듈 이름이 코드의 목적을 가장 강력하게 전달한다. `processData()` 대신 `calculateOrderTotalAndApplyDiscount()`처럼 의도가 드러나야 한다.
- **코드 구조가 의도를 드러낸다**: 하나의 함수/클래스는 하나의 책임만 가져야 한다(단일 책임 원칙). 복잡한 로직은 작은 단위로 분리한다.
- **타입을 명시적으로 선언한다**: 타입 추론에만 의존하지 않고, 호출자가 반드시 알아야 하는 정보는 타입으로 드러낸다.
- **매직 값은 상수/Enum으로 대체한다**: `if (status == 3)` 대신 `if (status == OrderStatus.SHIPPED)`.
- **도메인 용어(Ubiquitous Language)를 코드에 사용한다**: 기획자·도메인 전문가와 같은 언어를 코드에 반영한다.
- **조건·분기문은 읽기 쉽게 만든다**: Early Return, Guard Clause를 적극 활용하고, 복잡한 조건은 의미 있는 메서드로 추출한다.
- **주석은 ‘왜(Why)’를 설명한다**: “무엇(What)”은 코드로 표현하고, 주석은 비즈니스 규칙, 트레이드오프, 결정 이유를 담는다. 주석이 필요하다는 것은 코드 개선 신호로 받아들인다.

### 2. 작고 되돌릴 수 있는 단위로 변경한다

한 번의 커밋에는 하나의 명확한 의도만 담는다. 큰 리팩토링도 작은 단계로 나누어 항상 동작하는 상태를 유지한다.

### 3. 기존 컨텍스트의 관례를 따른다

기존 코드베이스에 합류했을 때는 개인 취향보다 프로젝트 내 일관성을 우선한다. 스타일 혼재는 어떤 나쁜 스타일보다 더 큰 문제를 만든다. 전체 개선이 필요하다면 팀 논의 후 일괄 적용한다.

### 4. 불확실할 때는 질문한다

요구사항이 모호하면 추측하지 말고 확인한다. 더 나은 대안이 보이면 적극적으로 제안한다.

### 5. 의사결정 원칙을 세워라

좋은 코드 구조만으로는 충분하지 않다. 팀이 함께 판단하고 실행하는 방식이 일관되지 않으면, 개인의 좋은 설계 습관은 코드베이스 전체로 이어지지 않는다.

**트레이드오프를 명시적으로 관리하라**

설계나 코드 리뷰, 리팩토링 판단을 내릴 때마다 “이번 결정에서 가장 중요한 평가 기준(지배축)은 무엇인가?”를 먼저 명확히 선언하는 것이 중요하다. 그리고 어떤 가치를 우선하고, 어떤 가치를 양보하는지 트레이드오프를 명시적으로 기록해야 한다.
예를 들어 “장기 유지보수성을 최우선으로 한다”고 결정했다면, 초기 개발 속도가 다소 느려지더라도 이를 받아들이고 그 이유를 팀과 공유해야 한다. 반대로 모든 평가축을 동시에 만족시키려다 모호한 타협을 하면, 나중에 누구도 책임지지 않는 애매한 코드가 만들어지기 쉽다.

설계 판단 시 내가 참조하는 8개 평가축은 다음과 같다.

| 평가축 | 주요 근거 및 출처 | 설명 |
|--------|------------------|------|
| **목적 적합성** | [Value-Focused Thinking (Ralph L. Keeney, 1992)](https://www.hup.harvard.edu/catalog.php?isbn=9780674931985) | 결정의 출발점인 “진짜 가치와 목적”을 최우선으로 삼아야 한다는 Keeney의 핵심 사상 |
| **제약 적합성** | Requirements Engineering (기능/비기능 요구사항), Systems Engineering | Functional & Non-functional requirements를 체계적으로 고려하는 전통적 접근 |
| **실현 가능성** | [INCOSE Systems Engineering Handbook의 Feasibility Analysis](https://www.incose.org/resources-publications/technical-publications/se-handbook/) | 기술적·경제적·운영적·스케줄 feasibility를 기술적·조직적 관점으로 압축 |
| **품질 영향** | [ISO/IEC 25019:2023 (Product Quality Model)](https://www.iso.org/standard/78177.html) | Maintainability, Reliability, Security, Performance 등 소프트웨어 품질 국제 표준 |
| **시간 효과** | [Technical Debt (Ward Cunningham, 1992)](https://c2.com/doc/oopsla92.html) | 단기 vs 장기 관점에서의 비용·가치 트레이드오프 |
| **위험 및 불확실성** | Decision Analysis / Decision Quality의 Uncertainty 평가 | 불확실성에 대한 명시적 체크 |
| **무결성** | [Architecture Decision Records (ADR, Michael Nygard)](https://www.cognitect.com/blog/2011/11/15/documenting-architecture-decisions) | 아키텍처 결정의 일관성과 장기적 무결성을 유지하기 위한 기록 수단 |
| **합의 가능성** | [Decision Quality의 Commitment to Action + Stakeholder Buy-in](https://sdg.com/decision-quality/) | 좋은 결정이라도 실행 의지와 합의가 없으면 실패한다 |

**결정은 기록하고 점진적으로 실행하라**

아무리 좋은 설계 판단을 내려도 실제로 실행되지 않거나 시간이 지나면서 일관성이 깨지면 그 결정은 의미가 없다. 중요한 결정은 ADR(Architecture Decision Record)로 기록하고, 팀 내 합의를 명확히 한 뒤 구체적인 실행 계획(담당자, 범위, 일정, 마이그레이션 전략)을 세워야 한다. 새로운 스타일을 도입하기로 결정했다면, 한 번에 모든 코드를 바꾸려 하지 말고 점진적으로 적용할 수 있는 전략을 함께 마련해야 한다.

## 설계 원칙

개발 원칙이 작업 방식과 의사결정 기준에 관한 것이라면, 설계 원칙은 실제 코드 구조와 표현에 관한 것이다. 코드의 장기적인 건강 상태를 결정짓는 기반이 된다.

### 1. 목적과 도메인에 집중하라

빈약한 데이터 모델(Anemic Domain Model)을 피하고, 도메인 객체가 비즈니스 규칙과 행동을 직접 책임지게 한다.

### 2. 작고 조합 가능하며 의도가 명확한 구조를 우선하라

함수는 작게 유지하고, 중첩을 최소화한다. 복잡한 로직은 의미 있는 작은 함수로 분리한다.

### 3. 추상화와 일반화는 중복이 실제로 드러난 후에 한다

동일한 코드가 3회 이상 반복될 때 추상화를 고려한다. 조급한 추상화는 오히려 유지보수성을 해친다.

### 4. 변화 용이성과 유지보수성을 최우선 가치로 삼아라

"3개월 후, 1년 후 다른 사람이 이 코드를 수정할 때 얼마나 쉽게 고칠 수 있을까?"를 가장 중요한 판단 기준으로 삼는다. 구현체가 아닌 추상 타입에 의존하고, 접근 권한을 최소화하며, 부수 효과를 줄인다.

### 5. 실현 가능성과 제약을 현실적으로 고려하라

팀 역량, 운영 부담, 스케줄 등을 무시한 과도한 기술 선택은 실패로 이어진다. 단순한 구조로 시작해 점진적으로 고도화한다.

## 실천

원칙이 '무엇을 해야 하는가'라면, 다음 다섯 가지는 '어떻게 실현하는가'에 해당한다. 리팩토링·네이밍·경계 조건·예외 처리·성능은 앞서 다룬 원칙들이 코드에 직접 드러나는 영역이다.

### 1. 리팩토링

리팩토링은 개발의 자연스러운 호흡이다. 기능 추가 전에 구조를 먼저 개선한다.

#### 리팩토링이 필요하다는 신호

경험이 쌓일수록 아래 신호를 보면 거의 본능적으로 리팩토링을 시작한다:

**Anemic Domain Model(행위 부족)**

 - 풍부한 도메인 행위가 아닌 CRUD 중심 설계 : 도메인 이벤트·행위 중심이 아니라 데이터 중심으로 모델링되어 있음
 - 도메인 로직이 퍼져 있음(God Class / God Service) : 도메인 규칙이 Application Service, Controller, Repository 등 여기저기 흩어져 있음
 - 도메인 지식이 인프라/기술 코드에 오염됨 : Persistence, UI, 외부 시스템 코드 안에 도메인 규칙이 섞여 있음

**Aggregate 경계(일관성 문제)**

 - Aggregate 경계가 깨짐 : Aggregate Root를 통해 접근해야 할 내부 객체들을 외부에서 직접 조작
 - Entity와 Value Object 구분이 없음 : 불변해야 할 개념을 가변 객체로 다루거나, 식별자 개념이 잘못 적용
 - 변경 시 도메인 일관성이 깨지기 쉬움 : 하나의 유스케이스를 수정할 때 여러 Aggregate나 Context를 동시에 건드려야 하고, 불변식이 쉽게 깨짐

**Bounded Context(의미/경계 문제)**

 - Bounded Context 경계가 모호하거나 없음 : 하나의 큰 모듈/서비스 안에 여러 도메인 개념이 뒤섞여 있음
 - 도메인 개념이 코드에서 점점 희미해지고, 기술적인 용어만 남을 때
 - Ubiquitous Language 위반 : 코드에 도메인 전문가와 공유하는 용어가 아닌 개발자 중심 용어가 많음

**공통 코드 스멜**

 - 동일한 코드가 3번 이상 반복될 때
 - 변수나 함수 이름이 그 의도를 충분히 설명하지 못할 때
 - 하나의 함수가 여러 책임을 가지기 시작할 때
 - 조건문 남용
 - 죽은 코드

#### 실천하는 태도

리팩토링을 할 때는 항상 작은 단위로, 그리고 테스트가 통과하는 상태를 유지하면서 진행한다. 큰 리팩토링도 여러 개의 작은 안전한 단계로 쪼개서, 중간중간 커밋을 만든다. 이렇게 하면 언제든지 되돌릴 수 있고, 리뷰어도 변경 의도를 쉽게 파악할 수 있다. 결국 좋은 개발자는 코드를 처음 작성하는 능력보다, **기존 코드를 지속적으로 건강하게 유지하는 능력**이 더 중요하다고 생각한다. 리팩토링은 그 건강을 유지하는 가장 강력한 도구 중 하나다.

### 2. 네이밍

이름을 짓는 일은 단순한 라벨링이 아니라, 중요한 설계 행위다. 변수, 함수, 클래스에 이름을 붙이는 순간, 나는 그 개념을 어떻게 이해하고 있는지 팀에게 선언하는 것이다. 좋은 이름은 코드를 읽는 사람에게 개념의 본질을 빠르고 정확하게 전달해야 한다. 나쁜 이름은 오히려 이해를 방해하고, 유지보수 비용을 증가시킨다.

**피해야 할 이름들**

`data`, `util`, `helper`, `manager`, `processor` 같은 이름은 거의 금기시한다. 이런 이름이 등장한다는 것은 그 개념의 역할과 책임이 아직 명확하게 정의되지 않았다는 강한 신호다. 개념 자체가 모호한 상태에서 이름을 붙이면, 나중에 반드시 후회하게 된다. 먼저 개념을 명확히 정리한 후에 이름을 지어야 한다.

**축약어는 최소화한다**

`usr`, `cfg`, `cnt`, `idx` 같은 축약어는 되도록 사용하지 않는다.  `user`, `config`, `count`, `index`처럼 원래 단어를 쓰는 것이 좋다. 타이핑하는 시간의 차이는 매우 작지만, 코드를 읽는 시간의 차이는 생각보다 크다.  코드는 한 번 작성되고 수십, 수백 번 읽힌다는 사실을 항상 기억한다.

**구현이 아닌 의도를 드러내는 이름을 사용한다**

이름은 구현 방식이 아니라, **그 코드가 하는 일의 의도**를 표현해야 한다. 동일한 개념에 여러 이름을 쓰지 않는다. `fetch`, `get`, `retrieve`, `load`가 같은 의미로 혼용되면 코드베이스 전체에서 패턴을 파악하기 어려워진다.

### 3. 경계 조건

경계 조건을 “예외적인 케이스”로 치부하는 것은 흔한 실수다. 재고가 없는 상황, 사용자가 로그인하지 않은 상황, 결제가 실패한 상황 — 이런 것들은 예외가 아니라 도메인이 정상적으로 다뤄야 할 **유효한 상태**다. 나는 경계 조건을 도메인의 일부로 명확하게 표현하려고 노력한다.

**명시적인 타입으로 경계 조건을 표현한다**

`null`, `None`, 빈 값을 암묵적으로 사용하는 대신, 가능한 한 명시적인 타입으로 표현한다.  
`Option<T>`, `Maybe<T>`, `Result<T, E>` 같은 타입을 활용하면 컴파일러가 경계 조건 처리를 강제하게 되어 숨겨진 가정이 사라진다.

Invalid state는 타입 수준에서 아예 존재할 수 없도록 설계하는 것이 가장 이상적이다. 런타임에 방어 코드를 여러 겹 쌓는 것보다, 그런 상태 자체가 생성되지 않게 만드는 설계가 훨씬 더 견고하다.

**컬렉션은 null 대신 빈 컬렉션을 반환한다**

호출하는 쪽에서 null 체크를 강요당하지 않고, 항상 안전하게 순회할 수 있게 된다. 이는 Optional이나 Result 타입을 사용하는 방향과 일맥상통한다.

**null과 빈 값은 분명히 구분한다**

null/None은 “값이 아직 설정되지 않았다”는 의미이고, 0, "", [] 같은 빈 값은 “값은 있지만 비어있다”는 의미다. 이 둘을 혼용하면 조건문에서 의도가 모호해지고, 숨겨진 버그가 발생하기 쉽다.

**오류를 컴파일 타임으로 끌어올린다**

가능한 한 많은 오류를 런타임에서 컴파일 타임으로 옮기는 것이 목표다. 타입 시스템을 적극 활용하고, 문자열 리터럴 대신 nameof(), keyof, stringify! 같은 타입 참조를 사용하는 것도 같은 전략이다. 이름을 바꿀 때 조용히 깨지는 대신, 컴파일러가 즉시 알려주게 만드는 것이다.

### 4. 예외 처리

경계 조건이 '어떤 상태를 허용할 것인가'의 문제라면, 예외 처리는 '허용되지 않은 상태가 발생했을 때 어떻게 반응할 것인가'의 문제다.

- 에러는 침묵해서는 안 된다. 의미 없는 기본값으로 덮어씌우는 것은 문제를 숨기는 행위다.
- assert는 프로덕션 입력 검증 수단이 아니다. 개발자 불변식(invariant)을 선언하는 용도로만 사용한다.
- 예외를 던질 때는 반드시 제대로 된 Exception 객체를 사용한다. 단순 문자열이나 객체를 throw하면 스택 트레이스가 사라져 디버깅이 매우 어려워진다.
- 타입 캐스팅은 런타임 검사를 동반해야 한다. 컴파일러가 타입 변환을 허용했다고 해서 런타임에도 안전하다는 보장은 없다. TypeScript의 `as`, Java의 다운캐스팅, Python의 `cast()` — 이것들은 컴파일러에게 "내가 책임진다"고 말하는 것이다. 그 책임은 런타임 검사로 뒷받침되어야 한다.
- switch/match 문에는 항상 default 케이스를 포함한다. default가 없으면 나중에 새 enum 값이나 케이스가 추가될 때 조용히 아무것도 하지 않는 버그가 생긴다. 이것은 "침묵하는 실패"의 전형이다. 처리할 내용이 없더라도 default를 명시적으로 두면 의도가 드러나고, 나중에 케이스가 추가될 때 컴파일러나 린터가 경고를 줄 수 있다.
- 예외(Exception)는 진짜 예외적인 상황에만 쓴다. 비즈니스 분기를 처리하기 위해 예외를 발생시키고 catch로 흐름을 제어하는 것은 두 가지 문제를 동시에 일으킨다. 첫째, 예외 객체 생성과 스택 캡처는 일반 조건문보다 훨씬 비싸다. 둘째, try-catch로 감싸진 흐름은 독자가 "이게 정상 경로인가, 오류 경로인가"를 판단하기 어렵게 만든다. 예측 가능한 상태는 조건문으로, 진짜 예외적인 상황만 예외로 표현한다.

### 5. 성능

성능 최적화는 보수적이어야 한다. Donald Knuth의 유명한 말이 지금도 여전히 정확하다고 믿는다.

> “Premature optimization is the root of all evil.”  
> (조급한 최적화는 모든 악의 근원이다)

- 성능 문제에는 **먼저 올바르게 동작하게 만든다** → **실제로 병목을 측정한다** → **측정된 병목 부분만 선택적으로 최적화한다** 순으로 접근한다. 측정 없이 직감으로 최적화를 시도하는 것은 대부분 엉뚱한 곳에 불필요한 복잡성만 추가하는 결과를 낳는다.
- 아래 사항들을 습관화한다.
  - **알고리즘 복잡도는 의식적으로 선택한다** : O(n²) 이상의 복잡도가 들어갈 때는 반드시 명확한 이유가 있어야 한다. 특히 대규모 데이터를 다룰 때는 더 신중해야 한다.
  - **N+1 문제는 가장 흔하고 위험한 패턴이다** : 반복문 안에서 DB 조회나 외부 API 호출을 하는 패턴은 성능 문제를 일으키는 대표적인 원인이다. 발견 즉시 개선 대상으로 삼는다.
  - **데이터는 필요한 것만 가져온다** : `SELECT *`는 거의 사용하지 않는다. 불필요한 컬럼을 가져오는 것은 네트워크 비용뿐만 아니라 인덱스 활용까지 방해한다.
  - **트랜잭션 범위는 최소한으로 유지한다** : 트랜잭션이 길어질수록 락 경합(lock contention)이 증가하고, 동시 처리 성능이 떨어진다. 필요한 최소 구간만 트랜잭션으로 감싼다.
  - **획득한 리소스는 반드시 해제한다** : DB 커넥션, 파일 핸들, 소켓 등은 사용 후 즉시 정리한다. 리소스 누수는 운영 환경에서 뒤늦게 발견되는 경우가 많아 디버깅이 매우 어렵다.
- 공유 가변 상태(shared mutable state)는 최대한 피한다. 동시성 문제 (race condition, deadlock 등)와 테스트 신뢰성 저하를 유발한다.

## 실패 패턴들

다음은 코드, 조직, 프로세스 수준에서 반복되는 실패 패턴들이다.

### 1. 바퀴의 재발명

- 인증(Spring Security, JWT 등), 로깅(Logback, Log4j, SLF4J, Winston, Serilog), 캐시(Redis, Ehcache, Caffeine, Guava Cache), 날짜, 알림, 국제화, 페이징, 유효성 검사, 상태관리 등은 라이브러리가 있는데 자체적으로 구현하는 것.
- DB 제약 조건으로 충분히 해결할 수 있는 것을 애플리케이션 코드에서 if문으로 검증하는 것.
- 기존에 참조하고 있는 라이브러리 등에서 사용중인 상수가 있는지 확인안하고 매직 숫자, 문자열을 남발하는 것.

### 2. Over Engineering

- DBA가 없는 팀에 복잡한 샤딩 + 멀티마스터 DB 구성을 처음부터 설계하는 것.
- 24시간 온콜 체계가 없는 팀인데 멀티 리전 고가용성 아키텍처를 도입하는 것.
- Kubernetes나 Kafka 운영 경험이 거의 없는 팀이 본격 도입하는 것.
- 상태가 ‘신청 → 승인 → 확정’ 정도로 단순한데 이벤트 소싱 + CQRS를 도입하는 것.
- 하루 500건 정도의 CSV 처리를 Kafka + Consumer Group으로 구성하는 것.
- 간단한 입력 검증(필수, 자리수, 정규식)에 Drools 같은 룰 엔진을 도입하는 것.
- SRE 전담 인력이 없는 팀에 SLO, Error Budget, Chaos Engineering을 도입하는 것.

### 3. 땜방식 대응

- 경계값 오류 (Off-by-one, 빈 배열, 최대값, 최소값, 길이, 인덱스 범위 초과)
- 에러가 발생해도 그냥 삼켜서 화면은 동작하게 만드는 것.(Empty list, 0, null, timeout 등)
- Controller나 Service에 도메인 로직을 직접 추가하는 것.
- 도메인 규칙을 무시하고 화면에서만 validation을 하는 것.
- null/nil/None 체크 안하거나 NullPointerException만 피하려고 if (x != null)을 단순 추가하는 것.
- 테스트가 깨지면 기대값을 실제 구현에 맞춰 수정하는 것.
- 타입을 안 쓰거나, Any/unknown/Object 남발 (특히 TS, Python)
- 하드코딩 (URL, API 키, 설정값, 매직 넘버)

### 4. 준비 안된 출발

- 요구사항이 아직 불명확한데 화면, API, DB를 먼저 만드는 것.
- 에러 발생 시 업무 처리가 정해지지 않았는데 예외 처리 코드를 먼저 작성하는 것.
- 권한 정책이 미정인데 UI에서만 숨기는 것.
- 데이터 이관 방침이 결정되기 전에 이관 스크립트를 먼저 작성하는 것.
- 성능 요구사항(건수, 응답시간)이 정해지지 않았는데 Redis 캐시를 처음부터 도입하는 것.

### 5. 너무 이른 추상화

설계 원칙 3번("추상화와 일반화는 중복이 실제로 드러난 후에 한다")과 같은 맥락이다. 조직·프로세스 수준에서 반복되는 구체 사례는 다음과 같다.

- 구현이 아직 하나밖에 없는데 Strategy 패턴을 도입하는 것.
- 검색 조건이 name과 status 두 개뿐인데 독자 DSL을 설계하는 것.
- 설정 항목이 거의 변하지 않을 것 같은데 미리 DB 기반 설정 관리 화면을 만드는 것.

### 6. 핵심 제약을 사후에 추가하는 역순 개발 관행

- CRUD 기능을 먼저 만든 뒤, 나중에 권한 체크를 추가하는 것.
- 데이터 갱신 기능을 만든 뒤에 감사 로그를 뒤늦게 추가하는 것.
- 삭제 기능을 만든 후에 “이력 관리가 필요하다”는 요구를 받고 대응하는 것.
- 동기 연동을 만든 후 재시도·멱등성(idempotency) 처리를 나중에 붙이는 것

### 7. "한 번 데였다고 다 금지하는" 반지성적 규칙

- TypeScript 도입을 “타입 관리할 사람이 없다”며 거부하고 계속 JavaScript로 개발하는 것.
- 과거에 빌드 자동화가 실패한 경험이 있다고 해서 CI/CD를 포기하고 수동 빌드·배포를 유지하는 것
- 과거 Lambda/Stream API로 버그가 난 적이 있다고 해서, 모든 반복문을 확장 for문으로 강제하는 것.
- Optional 사용을 금지하고, null 체크를 호출하는 쪽에 모두 떠넘기는 것.

### 8. 과거에 대한 과도한 배려

- 기존 API를 깨뜨리지 않으려고 신규 API와 기존 API를 병렬로 계속 유지하는 것.
- 오류 거부감 때문에 is_deleted, is_archived, is_suspended 같은 DB 칼럼이 계속 쌓이는 것.

## 안티 패턴 요약

안티 패턴은 코드 수준에서 반복되는 설계 문제다. 다음 패턴들은 개별 언어와 무관하게 반복적으로 나타난다.

| 안티 패턴 | 증상 | 해결 방향 |
|-----------|------|-----------|
| 성급한 추상화 | 중복이 없는데 인터페이스와 레이어가 넘침 | 3회 반복 후 추상화 |
| 빈약한 도메인 모델 | 서비스 레이어에 비즈니스 로직 집중 | 도메인 객체에 행동 부여 |
| 깊은 중첩 구조 | 조건문 4단계 이상 | 조기 반환, 메서드 분리 |
| 암묵적 경계 처리 | null 반환, 침묵하는 실패 | 명시적 타입과 에러 전달 |
| 테스트 없는 핵심 로직 | 리팩토링 시 회귀 발생 | 핵심 경로 테스트 먼저 |
| 측정 없는 성능 최적화 | 가독성이 나쁜 "최적화된" 코드 | 프로파일링 후 접근 |
| N+1 쿼리 | 반복문 내 DB 호출 | 배치 조회, JOIN 활용 |
| 무분별한 상속 계층 | 기반 클래스 수정이 파생 클래스를 깨뜨림 | 타입 기본 봉인, 합성 우선 |
| 스타일 혼재 | 한 파일 안에 두 가지 관례가 공존 | 기존 관례 따르거나 전체 일괄 변경 |
| 매직 넘버 사용 | 코드에 의미 불명의 숫자/문자열 리터럴 직접 등장 | 의미 있는 이름의 상수로 대체 |
| 예외를 제어 흐름으로 사용 | try-catch로 정상 분기를 표현 | 예측 가능한 상태는 조건문으로 처리 |
| 죽은 코드 방치 | 주석 처리 코드·미사용 함수가 코드베이스에 잔존 | 즉시 삭제, 이력은 git이 보관 |

LLM을 통해 AI와 함께 코드를 작성할 때, AI가 코드를 생성하는 기준이 명확해야 한다. "좋은 코드"의 정의가 모호하면 AI는 매번 다른 스타일의 코드를 만들어낸다. 이 기준을 AI 어시스턴트의 공통 규칙 파일에 두고 모든 프로젝트에 적용하면, AI와의 협업에서도 일관된 설계 판단이 나온다.

핵심은 단순하다. **읽기 쉽고, 도메인을 반영하며, 변경하기 쉬운 코드.** 이 세 가지를 유지하는 것이 언어를 불문하고 좋은 코드의 본질이다.
