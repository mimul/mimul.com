코딩 스타일 가이드는 보통 언어별로 존재한다. Python에는 PEP 8이 있고, Java에는 [Google Java Style Guide](https://google.github.io/styleguide/javaguide.html)가 있으며, Go에는 Effective Go가 있다. Rust 프로젝트를 시작하면 `rustfmt`와 Clippy가 스타일을 잡아준다. 이렇게 언어별 컨벤션은 잘 정비되어 있지만, 막상 여러 언어를 오가며 작업할 때 일관된 설계 철학이 없으면 코드는 언어만 다를 뿐 같은 문제를 반복한다.

LLM를 활용해 Java, Rust, TypeScript, Python, Go 등 여러 언어로 동시에 작업하면서 이 문제가 더 선명해졌다. AI가 코드를 생성할 때 적용할 수 있는 **언어 독립적인 설계 원칙**이 필요했다. 단순히 "중첩을 줄여라", "이름을 명확히 해라" 수준이 아니라, 왜 그렇게 해야 하는지 근거가 있는 원칙을 기술한다.

## 철학적 뿌리

이 코딩 스타일은 세 사람의 사상을 중심으로 구성된다.

**Kent Beck**은 XP(Extreme Programming)와 TDD를 통해 "동작하는 코드를 빠르게, 그리고 지속적으로 개선하라"는 관점을 제시했다. 그의 핵심 통찰은 소프트웨어 설계의 목표가 완벽한 구조를 한 번에 만드는 것이 아니라, **변경하기 쉬운 상태를 항상 유지하는 것**이라는 점이다.

**Martin Fowler**는 리팩토링을 체계화했다. 그에게 리팩토링은 기능을 추가하기 전에 코드를 정돈하는 별도의 단계가 아니다. 코드를 읽고 이해하고 수정하는 모든 과정에 자연스럽게 녹아 있어야 하는 **개발 리듬**이다.

**Eric Evans**는 DDD(Domain-Driven Design)를 통해 코드가 비즈니스 도메인의 언어를 반영해야 한다고 주장했다. 코드베이스가 도메인 전문가와 개발자가 함께 쓰는 언어로 작성될 때, 요구사항의 변화를 코드에 정확하게 반영할 수 있다.

이 세 관점은 겉으로는 다르지만 하나의 방향을 가리킨다. **코드는 읽히고, 이해되고, 변경될 것이다. 그 변경을 쉽게 만드는 것이 설계의 본질이다.**

## 핵심 원칙

다음 여섯 가지는 언어나 팀 규모와 무관하게 개발 태도와 의사결정 기준으로 삼는 원칙들이다. 나는 코드를 작성할 때 항상 이 원칙들을 가장 먼저 떠올린다. 이 원칙들은 경험을 통해 얻은, 실무에서 가장 중요한 기준들이다.

### 1. 변화 용이성을 최우선으로

코드를 작성할 때 가장 먼저 고려하는 것은 성능이 아니다. **“3개월 후, 혹은 1년 후에 누군가가 이 코드를 쉽게 수정할 수 있을까?”** 가 가장 중요한 질문이다. 성능도 물론 중요하지만, 그것은 실제 병목이 명확히 측정된 이후에 고민할 문제다. 설계 단계에서 미리 성능을 예측하며 복잡도를 높이는 것은 대부분의 프로젝트에서 오히려 독이 된다. 지금 당장의 성능 이득보다 미래의 유지보수 비용이 훨씬 더 크기 때문이다.

미래의 요구사항을 예측해 미리 추상화를 만드는 행위도 경계한다. 나는 **Rule of Three**를 철저히 따른다. 동일한 패턴이 세 번 반복되기 전까지는 추상화를 도입하지 않는다. 그 전에 만드는 추상화는 대개 잘못된 방향으로 코드를 묶어버리는 결과를 낳는다.

### 2. 코드는 의도를 명확하게 말해야 한다

짧은 코드가 좋은 코드가 아니다. 진짜 좋은 코드는 “무엇을 하는지”가 아니라 **“왜 존재하는지”** 를 분명하게 전달한다.

```rust
// 나쁜 예: 무엇을 하는지는 알겠지만 왜 하는지 모른다
if user.role == 1 && user.status == 0 {
    send_email(user);
}

// 좋은 예: 의도가 드러난다
if user.is_eligible_for_notification() {
    send_email(user);
}
```

주석을 달기 전에 먼저 코드 구조를 개선해야 한다. 주석이 필요하다는 것은 코드가 충분히 자기 의도를 설명하지 못하고 있다는 신호다. 명시적인 선언도 같은 맥락이다. 타입 추론이 편리하더라도, 호출하는 쪽에서 반드시 알아야 하는 정보는 명시적으로 드러내는 것이 좋다.

```typescript
// 반환 타입을 TypeScript가 추론에 맡김
function findUser(id: string) {
  return users.find(u => u.id === id);
}

// 반환 타입을 명시해 의도를 드러냄
function findUser(id: string): User | undefined {
  return users.find(u => u.id === id);
}
```

매직 넘버와 매직 스트링도 최대한 제거한다. 숫자와 문자열은 이름을 가진 상수로 만들어 의도를 명확히 한다.

```python
# 나쁜 예: 1609.344 가 무엇인지 알 수 없다
def mile_to_metre(mi: float) -> float:
    return mi * 1609.344

# 좋은 예: 의도와 단위가 드러난다
ONE_MILE_IN_METRES = 1609.344

def mile_to_metre(mi: float) -> float:
    return mi * ONE_MILE_IN_METRES
```

주석도 같은 원칙을 따른다. "주석보다 코드 구조 개선"이 우선이지만, 불가피하게 주석이 필요하다면 "무엇"이 아닌 **"왜"** 를 써야 한다. 코드를 읽으면 무엇을 하는지는 보인다. 코드로는 보이지 않는 것이 왜 그렇게 했는지다.

```python
# 나쁜 예: 코드가 이미 말하는 것을 반복
# 금액이 1000 이상이면 5% 할인
if total >= 1000:
    total *= 0.95

# 좋은 예: 코드로 보이지 않는 이유를 기록
# 2018년 VIP 캠페인 약정. 1000 이상 주문에 5% 할인 (기획서 #1234)
if total >= 1000:
    total *= 0.95
```

### 3. 도메인 언어를 코드에 반영한다

개발자와 도메인 전문가가 서로 다른 언어를 사용하면, 그 순간부터 지속적인 번역 비용이 발생한다. 요구사항에서 “주문 취소”라고 하면 코드에서도 cancelOrder처럼 일관되게 표현되어야 한다.
팀 내에서 합의된 언어를 코드, 변수명, 메서드명, 클래스명에 최대한 반영한다. 범용 유틸리티는 도메인 개념이 먼저 명확해진 후에 만들어야 한다.

### 4. 작고 되돌릴 수 있는 단위로 변경한다

변경은 언제나 작은 단위로 한다. 한 번의 커밋에는 하나의 명확한 의도만 담는다. 이렇게 해야 리뷰가 수월해지고, 문제가 발생했을 때 쉽게 되돌릴 수 있다. 큰 리팩토링을 할 때도 마찬가지다. 중간 중간 항상 동작하는 상태를 유지하면서, 각 단계마다 테스트를 통과하게 만든다.

### 5. 기존 컨텍스트의 관례를 따른다

새로운 프로젝트를 시작할 때는 내 스타일과 원칙을 마음껏 적용할 수 있다. 하지만 기존 코드베이스에 합류하거나, 이미 있는 파일을 수정할 때는 이야기가 다르다. 그 파일과 프로젝트에서 이미 사용 중인 관례가 내 개인 취향보다 우선이다.
코드베이스 전체에 두 가지 스타일이 혼재하는 것은, 어느 한쪽이 “더 나쁜” 스타일인 것보다 더 큰 문제다. 일관성은 읽기 쉬운 코드의 기본 조건이기 때문이다. 기존 스타일이 마음에 들지 않는다면, 팀과 충분히 논의한 후에 전체를 한 번에 바꾸는 방향으로 가는 것이 옳다.

### 6. 불확실할 때는 질문한다

요구사항에 모호한 부분이 있으면 절대 추측으로 채우지 않는다. 추측은 나중에 큰 수정 비용을 유발하는 잘못된 설계의 시작이다. 더 좋은 접근 방법이 보인다면 솔직하게 제안한다. “이렇게 하는 게 더 간단하지 않을까요?”라는 한 마디가 나중에 많은 시간을 절약해 준다.

## 설계 원칙

핵심 원칙이 작업 태도와 의사결정 기준이라면, 설계 원칙은 실제 코드 구조와 표현에 관한 것이다. 내가 실제로 코드를 설계하고 작성할 때 가장 중요하게 생각하는 원칙들이며, 코드의 장기적인 건강 상태를 결정짓는 기반이 된다.

### 1. 작고 조합 가능한 구조

함수는 하나의 책임만 가져야 한다. 조건문과 중첩의 깊이는 최대 2단계로 제한한다. 이 두 가지는 서로 밀접하게 연결되어 있다. 함수가 여러 책임을 가지면 자연스럽게 중첩이 깊어지고, 중첩이 깊어지면 흐름을 이해하기 어려워진다.

복잡한 조건문이 나오면 바로 의미 있는 메서드로 분리한다. 나는 "Tell, Don't Ask" 원칙을 좋아한다. 객체의 상태를 물어서 외부에서 결정하는 대신, 객체에게 직접 행동을 요청하는 방식이다.

```typescript
// Ask 방식: 외부에서 상태를 물어 결정
if (order.status === 'pending' && order.items.length > 0) {
  order.status = 'confirmed';
}

// Tell 방식: 객체에게 행동을 위임
order.confirm();
```

조건식이 복잡하다면, 메서드로 분리하기 전이라도 의미 있는 이름의 임시 변수로 먼저 분해한다. 이름을 붙이는 것만으로도 코드의 의도가 크게 개선된다.

```python
# 나쁜 예: 조건식이 무엇을 판단하는지 한눈에 들어오지 않는다
if (file.open(path, "w") is not None) and is_valid(path) or is_cached(path):
    process(file)

# 좋은 예: 임시 변수로 의도를 명시한다
file_is_accessible = (file.open(path, "w") is not None) and is_valid(path)
result_is_cached = is_cached(path)
if file_is_accessible or result_is_cached:
    process(file)
```

### 2. 도메인 모델에 행동을 담는다

빈약한 도메인 모델(Anemic Domain Model)은 초기에는 단순해 보이지만, 도메인이 복잡해질수록 큰 문제를 만든다. 데이터는 데이터 클래스에, 로직은 서비스 레이어에 분산되어 로직의 파악과 유지보수가 점점 어려워진다.
나는 가능한 한 데이터와 행동을 함께 둔다. 도메인 로직은 해당 객체 내부에 캡슐화하는 것을 선호한다. 이렇게 하면 도메인 개념이 코드에서 명확하게 드러나고, 관련된 로직이 한 곳으로 모인다.

### 3. 추상화는 중복이 드러난 이후에

추상화의 가장 중요한 목적은 중복 제거다. 중복이 명확히 드러나기 전에 추상화를 만들면 오히려 코드 탐색 비용만 증가한다. 미래의 확장을 대비해 미리 인터페이스나 팩토리를 만드는 행위는 지양한다. YAGNI(You Aren't Gonna Need It) 원칙을 철저히 따른다. 지금 필요하지 않은 것은 만들지 않는다.
같은 이유로 타입은 기본적으로 확장에 닫혀 있어야 한다. 상속이나 확장을 허용하는 것은 API를 공개하는 것과 같다. 필요가 명확해진 후에만 열어야 한다. 처음부터 열어두면 나중에 닫는 것이 훨씬 더 어렵다.

### 4. 접근 권한은 최소한으로 정의한다

클래스, 메서드, 변수의 공개 범위는 필요한 최소한으로 제한한다. 한 번 public으로 열어버리면 그것은 외부와의 연결이 된다. 나중에 변경하려면 모든 호출 지점을 확인해야 하는 부담이 생긴다.
새로운 멤버를 선언할 때는 항상 가장 좁은 범위(private)부터 시작한다. 범위를 넓히는 것은 언제든 가능하지만, 이미 넓게 열어둔 것을 다시 좁히는 것은 매우 어렵다.

### 5. 추상 타입으로 참조한다

변수를 선언할 때 가능한 한 구현 타입이 아닌 추상 타입(인터페이스나 추상 클래스)으로 선언한다. 구체적인 구현은 할당하는 오른쪽에만 드러나게 한다.

```java
// 나쁜 예: 구현 타입으로 선언하면 구현 교체 시 선언부도 변경해야 한다
ArrayList<User> users = new ArrayList<>();

// 좋은 예: 추상 타입으로 선언, 구현은 우변에만
List<User> users = new ArrayList<>();
```

구현을 `LinkedList`나 다른 자료구조로 교체할 때, 선언 타입이 `List`라면 선언부를 건드리지 않아도 된다. 이 습관이 쌓이면 구현 변경의 파급 범위가 자연스럽게 줄어든다. 이는 "변화 용이성"의 작고 구체적인 실천이다.

### 6. 변수는 사용 직전에 선언한다

변수의 선언 위치와 사용 위치가 멀어질수록 코드를 읽는 사람이 더 많은 문맥을 기억해야 한다. 따라서 변수는 처음 사용하는 지점 바로 위에서 선언하는 것을 원칙으로 한다.

```python
# 나쁜 예: 선언과 사용이 멀다
result = None
items = fetch_items()
filtered = [x for x in items if x.active]
# ... 다른 처리들 ...
result = calculate(filtered)

# 좋은 예: 필요한 순간에 선언
items = fetch_items()
filtered = [x for x in items if x.active]
result = calculate(filtered)
```

스코프를 최대한 좁게 유지하는 것도 중요하다. 블록 밖에서 사용되지 않는 변수는 블록 안에서 선언한다. 또한 하나의 변수를 여러 용도로 재사용하는 것도 피한다. 같은 이름이라도 의미가 달라지면 독자에게 혼란을 준다.

## 리팩토링은 개발의 리듬이다

리팩토링을 단순한 청소 작업으로 보지 않는다. 리팩토링은 개발의 자연스러운 리듬이자, 좋은 코드를 유지하기 위한 필수적인 호흡이다.

Martin Fowler가 말했듯이, 리팩토링은 **기능 추가와 교대로 이루어져야** 한다. 새로운 기능을 추가하려고 하는데 기존 코드의 구조가 그 기능을 받아들이기 어렵다면, 기능을 넣기 전에 먼저 구조를 개선한다. 이것은 별도의 “리팩토링 스프린트”로 미루는 것이 아니라, 매일의 개발 습관이 되어야 한다.

### 리팩토링이 필요하다는 신호

경험이 쌓일수록 아래 신호를 보면 거의 본능적으로 리팩토링을 시작한다:

- 동일한 코드가 3번 이상 반복될 때 (Rule of Three)
- 변수나 함수 이름이 그 의도를 충분히 설명하지 못할 때
- 하나의 함수가 여러 책임을 가지기 시작할 때
- 도메인 개념이 코드에서 점점 희미해지고, 기술적인 용어만 남을 때

이 신호들을 무시하고 기능을 계속 쌓아 올리면, 기술 부채가 빠르게 증가한다. 기술 부채는 복리로 불어난다. 처음에는 10분이면 고칠 수 있던 것이, 나중에는 며칠을 투자해야 하는 큰 작업으로 변한다.

### 죽은 코드는 과감하게 제거한다

주석 처리된 코드, 더 이상 사용하지 않는 함수, 오래된 설정 파일 등은 즉시 삭제하는 편이다.  

이런 죽은 코드가 코드베이스에 남아있으면 읽는 사람에게 불필요한 의문을 계속 던진다. “이 코드는 나중에 다시 쓰일까? 지금은 왜 남아있는 거지?”라는 생각이 들 때마다 인지 비용이 발생한다.

git이 모든 이력을 안전하게 기록하고 있다는 사실을 잘 알고 있다. 필요하면 언제든 과거로 돌아가 복구할 수 있기 때문에, 삭제를 두려워하지 않는다. 오히려 불필요한 코드를 방치하는 것이 코드베이스를 더럽히고, 팀 전체의 생산성을 떨어뜨린다고 믿는다.

### 실천하는 태도

리팩토링을 할 때는 항상 작은 단위로, 그리고 테스트가 통과하는 상태를 유지하면서 진행한다. 큰 리팩토링도 여러 개의 작은 안전한 단계로 쪼개서, 중간중간 커밋을 만든다. 이렇게 하면 언제든지 되돌릴 수 있고, 리뷰어도 변경 의도를 쉽게 파악할 수 있다.

결국 좋은 개발자는 코드를 처음 작성하는 능력보다, **기존 코드를 지속적으로 건강하게 유지하는 능력**이 더 중요하다고 생각한다. 리팩토링은 그 건강을 유지하는 가장 강력한 도구 중 하나다.

## 네이밍은 도메인의 번역이다

이름을 짓는 일은 단순한 라벨링이 아니라, 중요한 설계 행위다. 변수, 함수, 클래스에 이름을 붙이는 순간, 나는 그 개념을 어떻게 이해하고 있는지 팀에게 선언하는 것이다. 좋은 이름은 코드를 읽는 사람에게 개념의 본질을 빠르고 정확하게 전달해야 한다. 나쁜 이름은 오히려 이해를 방해하고, 유지보수 비용을 증가시킨다.

### 피해야 할 이름들

`data`, `util`, `helper`, `manager`, `processor` 같은 이름은 거의 금기시한다. 이런 이름이 등장한다는 것은 그 개념의 역할과 책임이 아직 명확하게 정의되지 않았다는 강한 신호다. 개념 자체가 모호한 상태에서 이름을 붙이면, 나중에 반드시 후회하게 된다. 먼저 개념을 명확히 정리한 후에 이름을 지어야 한다.

### 축약어는 최소화한다

`usr`, `cfg`, `cnt`, `idx` 같은 축약어는 되도록 사용하지 않는다.  `user`, `config`, `count`, `index`처럼 원래 단어를 쓰는 것이 좋다. 타이핑하는 시간의 차이는 매우 작지만, 코드를 읽는 시간의 차이는 생각보다 크다.  코드는 한 번 작성되고 수십, 수백 번 읽힌다는 사실을 항상 기억한다.

### 구현이 아닌 의도를 드러내는 이름을 사용한다

이름은 구현 방식이 아니라, **그 코드가 하는 일의 의도**를 표현해야 한다.

```python
# 구현 방식을 드러내는 이름
def get_user_from_db(user_id): ...
def loop_and_calculate_total(items): ...

# 의도를 드러내는 이름
def find_user(user_id): ...
def calculate_order_total(items): ...
```

동일한 개념에 여러 이름을 쓰지 않는다. `fetch`, `get`, `retrieve`, `load`가 같은 의미로 혼용되면 코드베이스 전체에서 패턴을 파악하기 어려워진다.

## 경계 조건은 도메인의 일부다

경계 조건을 “예외적인 케이스”로 치부하는 것은 흔한 실수다. 재고가 없는 상황, 사용자가 로그인하지 않은 상황, 결제가 실패한 상황 — 이런 것들은 예외가 아니라 도메인이 정상적으로 다뤄야 할 **유효한 상태**다. 나는 경계 조건을 도메인의 일부로 명확하게 표현하려고 노력한다.

### 명시적인 타입으로 경계 조건을 표현한다

`null`, `None`, 빈 값을 암묵적으로 사용하는 대신, 가능한 한 명시적인 타입으로 표현한다.  
`Option<T>`, `Maybe<T>`, `Result<T, E>` 같은 타입을 활용하면 컴파일러가 경계 조건 처리를 강제하게 되어 숨겨진 가정이 사라진다.

Invalid state는 타입 수준에서 아예 존재할 수 없도록 설계하는 것이 가장 이상적이다. 런타임에 방어 코드를 여러 겹 쌓는 것보다, 그런 상태 자체가 생성되지 않게 만드는 설계가 훨씬 더 견고하다.

### 컬렉션은 null 대신 빈 컬렉션을 반환한다

```python
# 나쁜 예: null 반환으로 호출자에게 null 체크 부담
def find_orders(user_id: str) -> list | None:
    if not user_id:
        return None

# 좋은 예: 빈 리스트로 "결과 없음"을 표현
def find_orders(user_id: str) -> list:
    if not user_id:
        return []
```

호출하는 쪽에서 null 체크를 강요당하지 않고, 항상 안전하게 순회할 수 있게 된다. 이는 Optional이나 Result 타입을 사용하는 방향과 일맥상통한다.

### null과 빈 값은 분명히 구분한다

null/None은 “값이 아직 설정되지 않았다”는 의미이고, 0, "", [] 같은 빈 값은 “값은 있지만 비어있다”는 의미다. 이 둘을 혼용하면 조건문에서 의도가 모호해지고, 숨겨진 버그가 발생하기 쉽다.

```python
count = 0  # 유효한 값: 재고가 0개
count = None  # 다른 의미: 재고를 아직 조회하지 않음

# 나쁜 예: 0과 None을 동시에 처리 — 의도가 모호하다
if not count:
    fetch_inventory()

# 좋은 예: 두 상태를 명시적으로 구분한다
if count is None:
    fetch_inventory()
```

### 오류를 컴파일 타임으로 끌어올린다

내가 추구하는 방향은 명확하다. 가능한 한 많은 오류를 런타임에서 컴파일 타임으로 옮기는 것이다.
타입 시스템을 적극 활용하고, 문자열 리터럴 대신 nameof(), keyof, stringify! 같은 타입 참조를 사용하는 것도 같은 전략이다. 이름을 바꿀 때 조용히 깨지는 대신, 컴파일러가 즉시 알려주게 만드는 것이다.

### 예외 처리 원칙

- 에러는 침묵해서는 안 된다. 의미 없는 기본값으로 덮어씌우는 것은 문제를 숨기는 행위다.
- assert는 프로덕션 입력 검증 수단이 아니다. 개발자 불변식(invariant)을 선언하는 용도로만 사용한다.
- 예외를 던질 때는 반드시 제대로 된 Exception 객체를 사용한다. 단순 문자열이나 객체를 throw하면 스택 트레이스가 사라져 디버깅이 매우 어려워진다.
- 예외는 진짜 예외적인 상황에만 사용한다. 정상적인 비즈니스 분기를 제어하기 위해 예외를 사용하는 것은 피한다.

```python
# 나쁜 예: assert는 -O 플래그로 실행 시 무시된다
def set_port(port: int) -> None:
    assert 1024 <= port <= 65535, f"Invalid port: {port}"

# 좋은 예: 명시적 예외로 항상 검증된다
def set_port(port: int) -> None:
    if not (1024 <= port <= 65535):
        raise ValueError(f"Port must be 1024–65535, got {port}")
```

```typescript
// 나쁜 예: 스택 트레이스가 없다
throw 'Invalid user id';
throw { message: 'not found', code: 404 };

// 좋은 예: 스택 트레이스가 보존된다
throw new Error('Invalid user id');
throw new NotFoundError('User not found');
```

타입 캐스팅은 런타임 검사를 동반해야 한다. 컴파일러가 타입 변환을 허용했다고 해서 런타임에도 안전하다는 보장은 없다. TypeScript의 `as`, Java의 다운캐스팅, Python의 `cast()` — 이것들은 컴파일러에게 "내가 책임진다"고 말하는 것이다. 그 책임은 런타임 검사로 뒷받침되어야 한다.

```typescript
// 나쁜 예: 런타임 검사 없는 단언
(x as User).getName();

// 좋은 예: instanceof로 검사 후 사용
if (x instanceof User) {
  x.getName();
}
```

switch/match 문에는 항상 default 케이스를 포함한다. default가 없으면 나중에 새 enum 값이나 케이스가 추가될 때 조용히 아무것도 하지 않는 버그가 생긴다. 이것은 "침묵하는 실패"의 전형이다. 처리할 내용이 없더라도 default를 명시적으로 두면 의도가 드러나고, 나중에 케이스가 추가될 때 컴파일러나 린터가 경고를 줄 수 있다.

```typescript
switch (status) {
  case 'active':
    enable();
    break;
  case 'inactive':
    disable();
    break;
  default:
    // 알 수 없는 상태 — 명시적으로 처리하지 않음을 선언
    throw new Error(`Unhandled status: ${status}`);
}
```

예외(Exception)는 진짜 예외적인 상황에만 쓴다. 비즈니스 분기를 처리하기 위해 예외를 발생시키고 catch로 흐름을 제어하는 것은 두 가지 문제를 동시에 일으킨다. 첫째, 예외 객체 생성과 스택 캡처는 일반 조건문보다 훨씬 비싸다. 둘째, try-catch로 감싸진 흐름은 독자가 "이게 정상 경로인가, 오류 경로인가"를 판단하기 어렵게 만든다. 예측 가능한 상태는 조건문으로, 진짜 예외적인 상황만 예외로 표현한다.

```python
# 나쁜 예: 예외를 제어 흐름으로 사용
def find_user(user_id: str) -> User:
    try:
        return user_map[user_id]
    except KeyError:
        return create_guest_user()  # 정상 분기인데 예외로 표현

# 좋은 예: 조건문으로 정상 흐름을 표현
def find_user(user_id: str) -> User:
    if user_id not in user_map:
        return create_guest_user()
    return user_map[user_id]
```

## 성능은 측정이 먼저다

나는 성능 최적화에 대해 매우 보수적인 입장을 가지고 있다. Donald Knuth의 유명한 말이 지금도 여전히 정확하다고 믿는다.

> “Premature optimization is the root of all evil.”  
> (조급한 최적화는 모든 악의 근원이다)

### 성능 문제에 접근하는 올바른 순서

1. **먼저 올바르게 동작하게 만든다**
2. **실제로 병목을 측정한다**
3. **측정된 병목 부분만 선택적으로 최적화한다**

측정 없이 직감으로 최적화를 시도하는 것은 대부분 엉뚱한 곳에 불필요한 복잡성만 추가하는 결과를 낳는다.

### 실무에서 지키는 원칙들

- **알고리즘 복잡도는 의식적으로 선택한다**  
  O(n²) 이상의 복잡도가 들어갈 때는 반드시 명확한 이유가 있어야 한다. 특히 대규모 데이터를 다룰 때는 더 신중해야 한다.

- **N+1 문제는 가장 흔하고 위험한 패턴이다**  
  반복문 안에서 DB 조회나 외부 API 호출을 하는 패턴은 성능 문제를 일으키는 대표적인 원인이다. 발견 즉시 개선 대상으로 삼는다.

- **데이터는 필요한 것만 가져온다**  
  `SELECT *`는 거의 사용하지 않는다. 불필요한 컬럼을 가져오는 것은 네트워크 비용뿐만 아니라 인덱스 활용까지 방해한다.

- **트랜잭션 범위는 최소한으로 유지한다**  
  트랜잭션이 길어질수록 락 경합(lock contention)이 증가하고, 동시 처리 성능이 떨어진다. 필요한 최소 구간만 트랜잭션으로 감싼다.

- **획득한 리소스는 반드시 해제한다**  
  DB 커넥션, 파일 핸들, 소켓 등은 사용 후 즉시 정리한다. 리소스 누수는 운영 환경에서 뒤늦게 발견되는 경우가 많아 디버깅이 매우 어렵다.

### 공유 가변 상태는 최대한 피한다

공유 가변 상태(shared mutable state)는 두 가지 큰 문제를 동시에 일으킨다:

- 동시성 문제 (race condition, deadlock 등)
- 테스트 신뢰성 저하

전역 가변 상태가 존재하면 테스트가 실행 순서에 따라 결과가 달라진다. 테스트 A가 상태를 변경한 채로 끝나면 테스트 B가 오염되어, 단독으로는 통과하지만 전체 스위트에서는 실패하는 현상이 자주 발생한다. 그래서 나는 전역 가변 상태를 최소화하려고 노력한다. 상태를 공유해야 하는 경우에도 불변 객체를 사용하거나, 명확한 소유권과 수명을 가진 구조로 설계한다.

## 안티 패턴 요약

좋은 설계 원칙을 이해하는 것만큼 나쁜 패턴을 인식하는 것도 중요하다. 다음 패턴들은 개별 언어와 무관하게 반복적으로 나타난다.

| 안티 패턴 | 증상 | 해결 방향 |
|-----------|------|-----------|
| 성급한 추상화 | 중복이 없는데 인터페이스와 레이어가 넘침 | 3회 반복 후 추상화 |
| 빈약한 도메인 모델 | 서비스 레이어에 비즈니스 로직 집중 | 도메인 객체에 행동 부여 |
| 깊은 중첩 구조 | 조건문 4단계 이상 | 조기 반환, 메서드 분리 |
| 암묵적 경계 처리 | null 반환, 침묵하는 실패 | 명시적 타입과 에러 전달 |
| 테스트 없는 핵심 로직 | 리팩토링 시 회귀 발생 | 핵심 경로 테스트 먼저 |
| 측정 없는 성능 최적화 | 가독성이 나쁜 "최적화된" 코드 | 프로파일링 후 접근 |
| N+1 쿼리 | 반복문 내 DB 호출 | 배치 조회, JOIN 활용 |
| 무분별한 상속 계층 | 기반 클래스 수정이 파생 클래스를 깨뜨림 | 타입 기본 봉인, 합성 우선 |
| 스타일 혼재 | 한 파일 안에 두 가지 관례가 공존 | 기존 관례 따르거나 전체 일괄 변경 |
| 매직 넘버 사용 | 코드에 의미 불명의 숫자/문자열 리터럴 직접 등장 | 의미 있는 이름의 상수로 대체 |
| 예외를 제어 흐름으로 사용 | try-catch로 정상 분기를 표현 | 예측 가능한 상태는 조건문으로 처리 |
| 죽은 코드 방치 | 주석 처리 코드·미사용 함수가 코드베이스에 잔존 | 즉시 삭제, 이력은 git이 보관 |

## 이 기준을 공통으로 쓰는 이유

이 코딩 스타일은 특정 언어의 컨벤션이 아니다. Java, Rust, TypeScript, Python, Go — 어떤 언어로 작업하더라도 적용 가능한 설계 철학이다.

언어별 공식 스타일 가이드는 별도로 존재한다. 이 글의 원칙과 함께 참조하면 언어 특유의 관용구와 포매팅 기준을 맞출 수 있다.

| 언어 | 공식 스타일 가이드 | 핵심 도구 |
|---|---|---|
| Java | [Google Java Style Guide](https://google.github.io/styleguide/javaguide.html) | Checkstyle, google-java-format |
| Python | [PEP 8](https://peps.python.org/pep-0008/) | Black, Flake8, Ruff |
| Go | [Effective Go](https://go.dev/doc/effective_go) | gofmt (빌드에 내장) |
| Rust | [Rust API Guidelines](https://rust-lang.github.io/api-guidelines/) | rustfmt, Clippy |
| TypeScript | [Google TypeScript Style Guide](https://google.github.io/styleguide/tsguide.html) | ESLint, Prettier |

Google Java Style Guide의 핵심만 짚으면: 클래스명은 `UpperCamelCase`, 메서드·변수는 `lowerCamelCase`, 상수는 `UPPER_SNAKE_CASE`, 들여쓰기는 2칸, 줄 길이는 100자다. 재정의 메서드에는 `@Override`를 항상 붙이고, 잡은 예외를 조용히 무시할 때는 반드시 주석으로 이유를 남긴다.

LLM을 통해 AI와 함께 코드를 작성할 때, AI가 코드를 생성하는 기준이 명확해야 한다. "좋은 코드"의 정의가 모호하면 AI는 매번 다른 스타일의 코드를 만들어낸다. 이 기준을 AI 어시스턴트의 공통 규칙 파일에 두고 모든 프로젝트에 적용하면, AI와의 협업에서도 일관된 설계 판단이 나온다.

핵심은 단순하다. **읽기 쉽고, 도메인을 반영하며, 변경하기 쉬운 코드.** 이 세 가지를 유지하는 것이 언어를 불문하고 좋은 코드의 본질이다.
